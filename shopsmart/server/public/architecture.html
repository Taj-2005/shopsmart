ShopSmart — Architecture & CI/CD

System design, deployment topology, and GitHub Actions workflows

System overview

ShopSmart is a monorepo eCommerce platform with a Next.js 16 frontend and an Express.js REST API. Authentication is cookie-based (httpOnly); protected routes enforce RBAC via middleware.

flowchart LR subgraph Client[Frontend] A[Next.js 16] B[React 19 Tailwind] end subgraph API[Backend API] C[Express] D[Prisma ORM] end subgraph Data[Data] E[(PostgreSQL)] end A --> B C --> D Client -->|HTTPS withCredentials| API D --> E

Deployment architecture

Production traffic flows from the browser to the Next.js app on Vercel, then to the Express API on Render; the API uses a PostgreSQL database.

flowchart TB subgraph Users[Users] U[Browser] end subgraph Frontend[Frontend Vercel] V[ShopSmart Web] end subgraph Backend[Backend Render] R[Express API] end subgraph DB[Database] M[(PostgreSQL)] end U -->|HTTPS| V V -->|api and cookies| R R --> M

Monorepo structure

Single repository containing client, server, and e2e packages with clear boundaries.

Request lifecycle (server)

Each API request passes through middleware before reaching the controller.

sequenceDiagram participant Req as Request participant CP as cookieParser participant CORS as cors participant Auth as authenticate participant Role as requireAdmin or requireSuperAdmin participant Ctrl as Controller Req->>CP: Populate req.cookies CP->>CORS: Validate origin credentials CORS->>Auth: Read accessToken Auth->>Role: Enforce role if admin routes Role->>Ctrl: Business logic Ctrl-->>Req: JSON response

Client-side auth flow

sequenceDiagram participant App as Next.js App participant API as Express API participant User as User User->>App: Request withCredentials App->>API: GET or POST with cookies alt 200 OK API-->>App: Response and Set-Cookie App-->>User: Render else 401 Unauthorized API-->>App: 401 App->>API: POST auth refresh with cookie alt Refresh OK API-->>App: New tokens App->>API: Retry request else Refresh fail App-->>User: Redirect to login end end
  • All API calls use withCredentials: true (Axios) or credentials: "include" (fetch).
  • On 401, the interceptor calls POST /api/auth/refresh and retries once; if refresh fails, redirect to /login.
  • Route protection uses ProtectedRoute and auth context in layouts.

CI/CD workflows (ShopSmart)

GitHub Actions workflows that build, test, and deploy ShopSmart. Triggered on push or PR to main or develop when shopsmart/** or the workflow file changes.

Build and test workflows

Server Build Build
build.yml

Trigger: Push to main or manual. Runs npm install and npm run build in shopsmart/server. Validates backend compiles.

Unit tests Test
unit-tests.yml

Trigger: Push/PR to main or develop when shopsmart/** changes. Runs server unit tests (Jest, exclude integration) and client unit tests in parallel. Uses DATABASE_URL, JWT, and repo variables.

Integration Test
integration.yml

Trigger: Push/PR to main or manual. Single job: lint, test, build in shopsmart/server. Matrix: Node 18, 20, 22.

Integration tests Test
integration-tests.yml

Trigger: Push/PR to main or develop when shopsmart/** changes. Runs server integration tests only with real DB (Prisma generate and db push).

E2E tests Test
e2e-tests.yml

Trigger: Push/PR to main or develop when shopsmart/** changes. Builds server and client, starts both locally, runs Playwright E2E from shopsmart/e2e against the running app.

Deploy workflow (ShopSmart)

Deploy ShopSmart Server to EC2 Deploy
deploy_shopsmart_server_to_EC2.yml

On PR: Build-only check. On push to main (when shopsmart/server/** or workflow changes): SSH to EC2, clone/update repo, copy .env from secret SHOPSMART_SERVER_ENV, run npm ci, Prisma migrate, build, start or restart API with pm2 under SHOPSMART_DEPLOY_PATH (e.g. /opt/shopsmart).

Frontend is deployed separately (e.g. Vercel). Other repo workflows: deploy_to_GitHub_pages.yml, deploy_to_EC2.yml.
